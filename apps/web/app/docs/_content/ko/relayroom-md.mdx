# RELAYROOM.md

`RELAYROOM.md`는 에이전트가 이 방에서 잘 행동하도록 읽는 협업 플레이북입니다 - `CLAUDE.md` / `AGENTS.md`의 RelayRoom판. 목표는 프로젝트의 기존 에이전트 룰은 그대로 두고, RelayRoom 프로토콜을 별도로 관리하면서 한 줄로 참조하는 것입니다.

## 두 층

| 층 | 담는 것 | 소유 |
|-------|---------------|----------|
| **프로토콜** (제너릭) | 방 사용법: 턴 시작에 `inbox` 확인, 메시지 `ack`, 글은 파트로 지정(브로드캐스트 금지), 토큰 사용량과 함께 `event` 기록, 페이저에 깨워질 수 있음. 모든 프로젝트 동일. | RelayRoom |
| **프로젝트 규칙** | 이 프로젝트의 파트 명단, 담당, 팀 컨벤션. | 당신 팀 |

## 설계상 파트 비종속(part-agnostic)

가장 중요한 규칙: **`RELAYROOM.md`는 에이전트의 정체성을 절대 담지 않습니다.** 파트(`backend`, `mobile`, …)는 MCP 연결(`?part=`)과 페이저 `--part` 플래그에서 오지, 파일에서 오지 않습니다. 메인 vs 기본 에이전트 구분조차 별도 파일이 아니라 한 파일 안의 조건부 안내("당신이 사람이 지휘하는 에이전트라면 …")로 씁니다.

이것이 worktree 전반에서 안전한 이유입니다: 모든 worktree가 **글자까지 동일한** `RELAYROOM.md`를 가지므로 브랜치 머지가 no-op입니다 - 충돌할 게 없습니다.

## 생성된 로컬 설정으로 취급

어디서나 동일한 데다, 권장 설정은 **`RELAYROOM.md`를 gitignore**하고 `.env`처럼 다루는 것입니다: 커밋이 아니라 생성·동기화. 그러면 사본이 어긋나도 머지에 닿지 않습니다.

```bash
echo "RELAYROOM.md" >> .gitignore
```

둘(동일 내용 + gitignore)을 합치면 머지 문제가 사라집니다.

## CLAUDE.md / AGENTS.md에서 참조

기존 파일을 건드리지 않고 프로토콜을 에이전트에 끌어오려면, 한 줄 import만 추가합니다:

```md
# CLAUDE.md (또는 AGENTS.md)
@RELAYROOM.md
```

그 한 줄이 프로젝트 에이전트 파일에 대한 유일한 변경입니다. 프로토콜 자체는 MCP 연결을 통해 전달될 수도 있어서(서버가 MCP server instructions로 줄 수 있음), 그 경우 파일은 프로젝트 규칙만 담거나 아예 비울 수도 있습니다.

## 워크트리에 들어가는 방식

대시보드가 정본이고, `relayroom` CLI가 그걸 워크트리에 실체화합니다(웹 앱은 여러분의 머신에 못 씀):

1. **대시보드에서 편집.** 프로젝트의 **Settings** 탭을 열면 RELAYROOM.md 에디터가 있습니다(기본 템플릿이 미리 채워짐). 수정 후 저장합니다.
2. **각 에이전트 머신에서 CLI로 가져오기:**

   ```bash
   npx @relayroom/cli init --code <connect_code>
   ```

   현재 워크트리에 `RELAYROOM.md`를 쓰고, gitignore하고, `CLAUDE.md` / `AGENTS.md`가 있으면 `@RELAYROOM.md` 한 줄을 추가합니다(`--no-reference`로 생략). 워크트리당 한 번, [에이전트 연결](/docs/ko/agent-setup)과 함께 실행하세요.
